## 一、项目规划与架构 Agent

建议命名：

```text
project-architect
```

### 核心职责

负责把模糊需求变成可以直接开发的任务：

- 梳理需求目标
- 判断功能范围
- 设计页面结构
- 确定 Server Component 与 Client Component 边界
- 确定目录、模块和数据流
- 拆分开发任务
- 编写验收标准
- 记录重要技术决策
### 约束

这个 Agent：

- **不直接进行完整功能编码**
- 不随意添加依赖
- 不把未来需求提前实现
- 不修改已经确认的业务目标
- 必须明确“本次不做什么”

### 适合处理

- 新功能规划
- 技术选型
- 项目初始化
- 大规模重构
- 模块边界设计
- 复杂 Bug 的处理方案

## 二、功能交付 Agent

建议命名：

```text
feature-builder
```

### 核心职责

根据任务说明完成代码交付：

- 阅读任务卡和项目规则
- 定位现有代码
- 实现页面和组件
- 完成响应式布局
- 处理交互与动画
- 补充必要测试
- 执行自检
- 输出修改说明

### 适合处理

- 页面开发
- UI 优化
- 组件封装
- 动画实现
- API 或 Server Action
- Bug 修复
- 小规模重构


## 三、质量验收 Agent

建议命名：

```
quality-reviewer
```

这个 Agent必须与开发 Agent 相互独立。

### 核心职责

负责判断功能是否真正完成，而不是只判断“代码能不能运行”：

- 对照需求验收
- Review Git Diff
- 检查架构边界
- 检查 TypeScript 类型
- 检查 Server/Client Component 使用
- 检查响应式和交互
- 检查 SEO、可访问性和性能
- 执行测试与构建
- 输出通过或驳回结论

### 缺陷级别

|等级|含义|是否阻塞交付|
|---|---|---|
|P0|页面不可用、构建失败、数据安全问题|是|
|P1|主要功能错误、验收项未完成|是|
|P2|维护性、体验或性能问题|一般不阻塞|
|P3|风格和细节建议|否|

### 约束

这个 Agent：

- 不重新定义需求
- 不大规模重写开发代码
- 不用个人偏好代替项目规范
- 必须指出具体文件和问题
- 必须区分“阻塞项”和“优化项”
- 没有证据不能判定失败
- 不允许只回复“代码看起来没问题”
