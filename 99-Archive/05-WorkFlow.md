1. 定义 `AGENTS.md` ，可以在 Reddit 中找到大量帖子
	- 甚至可以维护一份 `/AGENTS-META/` ，针对不同场景切换规则。

> 有人把它比作"给 AI 写的员工手册" —— 你不可能每次开会都从头解释公司规矩，写一份文档让它自己读就行了。

2. 先定义边界，再讨论能力
	- "Codex 不稳定"，其实不是能力问题，而是边界没定清楚。
3. 它适合做执行层，不适合做判断层
	- 涉及产品判断、架构取舍、模糊需求的澄清、审美层面的打磨，你还是得自己来。
	- 人定方向，AI 干活。你不应该问 Codex "这个功能该不该做"或者"用户会不会喜欢这个交互"，这些是你的事。
	- 它是一个执行力很强但没有产品直觉的队友。你给它清晰的指令，它能帮你省大量时间；你给它模糊的方向，它可能帮你制造大量问题。
4. 真正值得投入时间的是 workflow design，不是 prompt engineering
	- 让 Codex 在一个隔离分支上干活，改完之后我 review diff，满意了再 merge。这样即使它搞砸了，我的主分支也不受影响。
5. 不要追求完全信任，要追求稳定可控
	- 我现在并不觉得自己比以前更"信任" Codex 了。我只是更清楚什么任务适合直接交给它、什么任务必须分阶段、什么地方一定要人工把关。
	- 怎么设计一个值得信任的流程
	- 你不信任 Codex 本身没关系，但你可以信任一个包含了计划审核、测试验证、分支隔离、人工 review 的流程 —— Codex 只是这个流程里的执行环节。


### Reddit 社区里沉淀下来的几个共识

把我看过的讨论归纳一下，社区里对 Codex 的认知大概收敛到这几点：

- **它是一个执行力很强的后台工作者**，不是聊天助手。越来越多的人把它当成队列任务处理器用 —— 早上排任务，白天它跑，你做别的，晚上 review

- **AGENTS.md 和规则文件是标配**，不是可选项。用过的人基本都不会再回到每次手写 prompt 的状态

- **长任务必须配流程**。自动化、review 代理、后台任务这些功能都有用，但前提是你自己先把验证和回滚机制搭好

- **讨论的重心已经从"模型聪不聪明"转向了"工作流怎么设计"**。哪个模型好用、prompt 怎么写最优，这些话题的热度明显在下降


还有一个有意思的现象：很多人不只用 Codex 一个工具。常见的搭配是 Codex/Claude Code 做重活和后台任务，Cursor 做日常交互式编码，Copilot 做行内补全。工具之间各取所长，而不是 all-in 一个。


### 我现在对 Codex 的判断

一句话：**我更倾向于把 Codex 看成一个需要被设计进工程流程里的执行系统，而不是一个只靠 prompt 驱动的智能助手。**


# Reddit 总结

比较稳定的共识是：

> `AGENTS.md` 不是一份“AI 开发百科”，而是项目的导航页、硬约束和验收入口。

问题不在于有没有 `AGENTS.md`，而在于内容是否必要、具体且与项目相关。



## Reddit 中最通用的规则

### 保持短小，只写项目特有信息

优先写：

- 项目的特殊约束
- 不容易从代码中推断的设计决策
- 精确的启动、检查和测试命令
- AI 经常犯的重复性错误
- 必须复用的组件、模式和工具

不要写：

- “编写干净、优雅、可维护的代码”
- TypeScript、Vue、React 等基础常识
- 能通过 `package.json` 或代码搜索直接发现的信息
- 大段技术教程和完整架构说明

### 使用明确、可验证的指令

推荐：

- 修改源代码后运行 `pnpm typecheck`。
- 修改业务逻辑后运行对应单元测试。
- 不允许通过删除断言、跳过测试或放宽校验使测试通过。


不推荐：

- 保证代码没有问题。
- 注意代码质量。
- 尽量完善测试。

> 有效模式是：**命令、触发条件、预期结果**必须明确，而不是依赖 Agent 自己解释“最佳实践”。


### 限制修改范围

比较通用的约束是：

- 只修改完成当前任务所需的文件。
- 不进行无关重构、格式化或顺手优化。
- 优先产生小而可审查的 diff。
- 保持项目已有代码风格。
- 未经明确要求，不重命名、移动或删除文件。

> 这类规则可以减少 Agent 把一个局部需求扩展成大范围重构。

### 明确定义完成标准

一个任务不能以“代码写完”为结束，而应至少满足：

```markdown
- 功能符合需求和验收条件。
- 相关 lint、类型检查、测试或构建已经执行。
- 失败项已经修复，或在结果中明确说明。
- 最终说明修改了什么、为什么修改、如何验证。
```

测试、Lint、类型检查等自动化守卫通常比纯自然语言规则更可靠；Reddit 用户也普遍建议把 `AGENTS.md` 与测试、分析器、格式化工具和 Git Hook 配合使用。


### 写清楚安全边界

适合放入通用规则的安全约束：

- 不输出或提交密钥、Token、密码和私有配置。
- 不修改生产环境数据。
- 不执行 `force push`、`DROP`、批量删除等破坏性操作。
- 外部 API 默认只读。
- 涉及不可逆操作时，先说明影响并等待明确授权。
- 不自行安装全局依赖或修改宿主机环境。

但真正的权限限制应由沙箱、审批策略和 CI 实现，不能只依赖 Markdown 中的一句“禁止”。


### 把 `AGENTS.md` 作为导航层

当内容开始变多时，不要持续向根文件堆积，而应改成：

```mk
- 架构说明：`docs/architecture.md`
- UI 规范：`docs/ui-guidelines.md`
- 代码审查：`docs/code-review.md`
- 任务规划：`docs/task-workflow.md`
```

Agent 根据任务按需读取对应文件。这种“渐进式加载”比每次把全部说明注入上下文更节省 Token，也更不容易发生规则干扰。


### 大型项目使用目录级规则

根目录负责全局约束：

```
AGENTS.md
```

子模块负责局部规则：

```
apps/web/AGENTS.mdservices/api/AGENTS.md
```

距离当前工作目录更近的规则优先，适合 Monorepo 或前后端规范差异较大的项目。官方开放格式和 Codex 文档都支持这种分层方式。


### 根据真实错误迭代，而不是提前设想全部规则

比较实用的维护原则是：

> Agent 第一次犯错时纠正；同类错误再次出现时，再写进 `AGENTS.md`。

这样规则来自真实摩擦，而不是预先生成几百行“也许有用”的约束。OpenAI 当前的 Codex 实践建议也是：从最少规则开始，重复错误或重复 PR 反馈出现后再补充。

### 关于文件长度

Reddit 上常见建议包括：

- 控制在 200 行以内
- 200～300 行是较舒适范围
- 超过 500 行容易产生规则干扰
- 有人只按照 32 KiB 上限控制

这些都只是经验值，**不存在统一的 200 行标准**。


建议是：**初版控制在 50～100 行，超过后开始拆分。**


## 推荐的通用初版

```
# AGENTS.md

## Project Goal

简要说明项目解决什么问题，以及当前主要开发目标。

## Source of Truth

- 当前任务和验收标准以 `task.md` 为准。
- 项目架构说明位于 `docs/architecture.md`。
- UI 规范位于 `docs/ui-guidelines.md`。
- 当文档与现有代码冲突时，先说明冲突，不要自行猜测。

## Scope

- 只修改完成当前任务所需的文件。
- 不进行无关重构、格式化、重命名或依赖升级。
- 优先采用最小、清晰、可审查的修改。
- 保持现有目录结构、代码风格和命名习惯。

## Workflow

1. 阅读任务、验收标准和相关代码。
2. 明确影响范围及实现方案。
3. 完成功能实现。
4. 执行相关检查。
5. 检查最终 diff 并汇报结果。

复杂或跨模块任务先给出简短计划；
局部且明确的修改可直接执行。

## Commands

- Install: `pnpm install`
- Development: `pnpm dev`
- Lint: `pnpm lint`
- Type check: `pnpm typecheck`
- Test: `pnpm test`
- Build: `pnpm build`

只执行与本次修改相关的检查；
无法执行时必须明确说明原因。

## Implementation Rules

- 优先复用已有组件、工具函数和数据结构。
- 不重复实现项目中已经存在的能力。
- 不添加没有实际恢复逻辑的 `try/catch`。
- 不通过静默降级或虚假默认值隐藏错误。
- 不使用 `any` 绕过能够明确表达的类型问题。
- 不删除或弱化测试来使检查通过。

## Safety

- 不读取、输出或提交密钥和敏感信息。
- 不执行破坏性 Git、数据库或远程 API 操作。
- 未经明确要求，不修改生产配置和部署流程。
- 对不可逆操作，先说明命令、影响和恢复方式。

## Definition of Done

任务完成必须满足：

- 实现符合任务和验收标准。
- 相关 lint、类型检查、测试或构建已执行。
- 没有未经说明的失败项。
- 最终汇报修改内容、涉及文件和验证结果。

## Maintenance

只有当规则适用于多数任务，或同类错误重复出现时，
才将其加入本文件；任务专用流程应放入独立文档。
```


**`task.md` 管当前任务，`AGENTS.md` 管长期规则，`docs/` 管按需知识。**
它比把需求、架构、工作流、角色定义全部塞入一个超长文件更稳定，也更符合 Reddit 当前比较成熟的实践。
