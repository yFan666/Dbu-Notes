## 1. 本周高频提问主题

### 1.1 AI 协作边界与防捏造

* 高频问题：

  * 参考页面或示例里没有的逻辑，AI 不能自己补。
  * 不确定时是否应该先问，而不是擅自决定。
  * 如何让 AI 按证据来源实现，不编字段、接口、权限、业务规则。

* 背后反映的核心需求：

  * 希望 AI 从“会生成内容”变成“稳定可靠的项目协作执行器”。
  * 真正关心的是可控、可验证、少返工，而不是 AI 做得更多。

* 可沉淀经验：

  * 没有证据就不要实现；必须先确认来源、标注待确认，或向用户提问。

### 1.2 最小范围修改与任务边界

* 高频问题：

  * 只实现当前需求，不要修改其他代码。
  * 小型调整不要启动完整复杂流程。
  * 不要顺手重构、抽组件、扩大影响范围。

* 背后反映的核心需求：

  * 希望 AI 的执行边界更稳定，避免“为了解决小问题引入大变化”。

* 可沉淀经验：

  * 默认只改当前需求直接相关内容；任何范围扩大都要有明确理由。

### 1.3 知识库结构与长期沉淀

* 高频问题：

  * AGENTS 如何建立初始规则。
  * 旧知识库如何迁移到 Dbu Notes。
  * 什么内容进入 Inbox、Knowledge、Projects、Records、Archive。
  * 写过的任务和回答如何形成复用。

* 背后反映的核心需求：

  * 想把零散聊天、项目经验、提示词、工作日志变成一个 AI 能接续、人也能复用的个人系统。

* 可沉淀经验：

  * 知识库不是存东西，而是恢复上下文；记录重点应是“为什么”和“下一步”。

### 1.4 Skill / Agent / MCP 的封装与使用

* 高频问题：

  * 高频协作规则如何封装成 skill。
  * skill 正文能不能用中文。
  * Codex 为什么没有识别项目里的 skill。
  * 如何在其他电脑复用 skill。

* 背后反映的核心需求：

  * 希望把重复教 AI 的流程，变成可安装、可迁移、可版本管理的能力包。

* 可沉淀经验：

  * Skill 适合封装流程，MCP 适合连接工具；仓库保存 skill 源文件，全局目录用于实际识别。

### 1.5 工作日志、复盘与完结标识

* 高频问题：

  * 工作日志如何写得更有用。
  * 每天任务如何和 No 编号关联。
  * 复盘应该短，但必须给下一步。
  * 所有事情都需要完结标识。

* 背后反映的核心需求：

  * 希望减少注意力挂账，让任务、想法、工作、休息都有清楚边界。

* 可沉淀经验：

  * 记录只保留“事实 / 判断 / 下一步”；没有下一步，就标记完成、暂停、归档或不再追踪。

### 2.2 AI 协作与 Agent 管理

本周 AI 协作的核心变化是从“规则散落在对话里”，转向“规则进入 AGENTS、知识库和 skill”。

已形成的关键规则：

* 最小范围修改。
* 参考页面 / 文档优先。
* 不确定先问，不要捏造。
* 完成后按需求逐项验证。
* 全局记忆只放跨项目稳定偏好，项目细节进入项目上下文或知识库。
* 跨线程总结不能只看当前聊天，要读文件、项目记录、提交记录和已有沉淀。

Agent 管理方向：

* 优先做低风险、贴近知识库的 Agent，例如知识库问答 Agent、Inbox 整理 Agent。
* 自动日报 Agent 和前端代码审核 Agent 先保留为想法，等结构和标准稳定后再推进。
* Skill 的定位被明确为“流程级复用”，MCP 的定位是“工具和数据源连接”。

### 2.3 提示词与创作模板

本周提示词主要形成了三类结构。

第一类：AI 回复风格模板。

* 结论先，最多 3 条可执行动作。
* 日常问题 1-3 句，只答所问。
* 只有用户要求展开、细讲、写方案、留档、改代码、端到端时才加长。

第二类：项目 / 需求执行模板。

* 需求要包含目标文件、参考页面、需求点、限制、验收检查。
* PM 需求文档要补齐页面信息、需求目标、功能说明、接口、字段、交互和验收标准。
* 需求完成后应记录代码定位、实现改动、验证结果、遗留问题。

第三类：设计 Prompt 框架。

* 固定风格系统。
* 可变主题输入。
* 语义转译机制。
* 元素融合规则。
* 风格边界。
* 质量验收。

可复用结构：

```text
最终作品 = 固定风格系统 × 主题语义 × 视觉转译 × 构图控制 × 边界约束 × 验收标准
```

## 4. 本周形成的方法论

### 证据边界项目工作流

* 适用场景：
  * 业务需求实现、问题排查、参考页面仿照、需求文档整理、跨线程复盘。
* 核心原则：
  * 先锁范围，再找证据；不确定先问；结束前按原始需求验证。
* 使用方式：
  * 执行前列参考源、照搬点、差异点、待确认、不应照搬；执行后列已处理、验证、待确认。
* 一句话总结：
  * 没有证据就不要实现，超出范围就先说明。

### AI Native 知识库方法论

* 适用场景：
  * 整理工作、项目、提示词、复盘和旧知识迁移。
* 核心原则：
  * 记录“为什么”，不是只记录“做了什么”。
* 使用方式：
  * 临时内容先进 Inbox，每周蒸馏；可复用经验进入 Knowledge；项目过程进入 Projects；时间节点进入 Records。
* 一句话总结：
  * 知识库是恢复上下文的系统，不是材料堆。

### SDD 规范驱动开发

* 适用场景：
  * AI Coding、需求实现、复杂任务拆解。
* 核心原则：
  * 先明确目标、上下文、边界、步骤、验收标准，再让 AI 执行。
* 使用方式：
  * 按“任务说明 → 上下文读取 → 计划拆解 → 执行修改 → 验证结果 → 沉淀复盘”推进。
* 一句话总结：
  * 让氛围变成结构，让 AI 协作可控、可验证、可复盘。

### 三分钟处理方法

* 适用场景：
  * 外界反馈、情绪波动、反复思考、注意力被占用。
* 核心原则：
  * 区分成长问题和面子问题；能行动就行动，不能行动就结束。
* 使用方式：
  * 记录事件、感受、事实、原因、下次方案、认知，然后回到主线。
* 一句话总结：
  * 解决问题，提炼模型；其余反复思考都是情绪消费。

### 完结标识规则

* 适用场景：
  * 任务、想法、工作、休息、复盘、项目阶段。
* 核心原则：
  * 每件事都需要完成、暂停、归档、下一步已记录，或明确不再继续。
* 使用方式：
  * 每次记录结尾写清下一步；没有下一步就给出完结状态。
* 一句话总结：
  * 没有完结标识的事情，会持续占用注意力。

## 5. 本周可沉淀到知识库的内容

### 5.1 项目经验

* 业务需求实现前必须确认字段、接口、权限、校验和参考页面来源。
* 工作日志中的每条任务要保留 No 编号，方便和任务池、需求系统反查。
* DbuuuuuBlog 不应先追求功能多，而应先定规则、目录、内容模型、首页闭环和验收标准。
* RAG 项目要先提升 Markdown 文档质量，再做检索和 Agent 自动化。

### 5.2 AI 协作经验

* AI 协作的核心不是让 AI 多做，而是少猜、少扩展、按证据闭环。
* 全局记忆只放稳定偏好和通用工作方式，项目细节进入项目上下文。
* Skill 适合封装重复流程，MCP 适合连接外部能力。
* 项目内 skill 适合版本管理，全局 skills 目录才用于 Codex 实际识别。

### 5.3 提示词模板

* 业务需求模板：目标文件、参考页面、需求点、限制、完成后检查。
* PM 需求模板：页面信息、需求目标、功能说明、接口字段、交互规则、验收标准。
* UI 方向模板：目标感受、参考图、必须保留、不要出现、当前阶段。
* 复盘模板：事实、判断、下一步。
* 设计 Prompt 框架：固定风格系统 × 主题语义 × 视觉转译 × 构图控制 × 边界约束 × 验收标准。

### 5.4 个人成长经验

* 复盘要短，但必须给下一步。
* 所有事情都需要完结标识。
* 面子受损不等于实力受损。
* 事情发生后只做两件事：解决问题、提炼模型。
* 想法要从悬空状态变成小闭环，至少有下一步或暂停理由。

### 8.3 全部线程共同主题

所有已落盘线程共同指向一个主题：

> 把 AI 从临时问答工具，变成有规则、有证据、有上下文、有验收标准的长期协作系统。

当前最需要继续做的不是新增更多规则，而是让这些规则在真实项目里反复验证、删减、稳定下来。
