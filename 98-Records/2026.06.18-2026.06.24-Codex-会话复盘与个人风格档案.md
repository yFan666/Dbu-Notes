## 3. 执行经验复盘

### 3.1 需求动作边界不清时直接执行

**导致的问题**

- 把“总结本周工作”理解成“整理并归档”，周内提前移动文件。
- 批量补笔记属性时，字段集合尚未确认，后续又补 `created / updated`。
- 目录编号和重命名开始前，没有先统一确认范围、编号规则和链接影响。

**最终正确做法**

1. 先区分动作：讨论、计划、记录、修改、迁移、归档。
2. 批量操作前提供预览，包括影响数量、命名规则、链接变化和不处理范围。
3. 用户确认后再执行；周内只总结，周日才做周总结和归档判断。

**适用场景**

- 知识库整理、批量属性、目录迁移、重命名、归档和自动化任务。

### 3.2 只照搬来源项目，没有对齐目标项目

**导致的问题**

- 安岳附页迁移时，来源项目中的字段联动被直接带入目标项目，形成未经确认的扩展。
- 只看某次来源提交，容易遗漏后续修正，也容易把无关改动一起同步。

**最终正确做法**

1. 先列来源提交的新增内容和后续修正。
2. 再对比目标项目已有同类页面和业务口径。
3. 建立字段、类型、联动、接口和交互差异表。
4. 只迁移当前需求需要的内容；差异项交给用户确认。

**适用场景**

- 跨仓库迁移、复制页面、参考旧项目、同步分支功能、复用提交记录。

### 3.3 AI 自称“已完成”或“逻辑闭环”

**导致的问题**

- 没有测试、运行结果或用户回归时，完成结论缺少证据。
- 接口字段、后端值和权限来源容易被合理化，而不是被验证。

**最终正确做法**

完成状态只接受以下证据之一：

- 代码差异与需求逐项对应。
- 类型检查、构建、测试或运行验证通过。
- 接口文档、现有代码或用户输入能够反查。
- 用户明确验收通过。

缺少证据时标记为“待验证”，不能写成“已完成”。

**适用场景**

- 业务功能、接口联调、权限、打印导出、表单校验、UI 验收。

### 3.4 长线程压缩后继续依赖模糊上下文

**导致的问题**

- 压缩或长时间续跑后，部分属性、约束和后续修正可能丢失。
- 新线程容易沿用旧线程中的 AI 推断，而不是用户确认内容。

**最终正确做法**

项目内统一使用：

```text
00-线程接力/
└── EPxx-主题/
    ├── 线程清洗总结.md
    └── 新线程交接文档.md
```

交接材料必须区分：

- 已确认事实
- 已确定规则
- 当前有效结论
- 待验证内容
- 不可信 / 应丢弃内容
- 下一步任务

**适用场景**

- 自动压缩前后、跨天继续、长线程、多人或多 Agent 接力。

### 3.5 只看当前线程做全局总结

**导致的问题**

- 漏掉 smarthos-cdms-manager、smarthos-front 等其他线程的实际工作。
- 文件修改时间无法代表完整的工作过程。

**最终正确做法**

全局总结按以下证据顺序执行：

1. 会话索引和相关线程元数据。
2. 目标日期内的用户请求和执行结果片段。
3. 项目 Git 变化、构建和测试记录。
4. Dbu Notes 当天或本周记录。
5. 明确标出无法读取或证据不足的部分。

**适用场景**

- 每日总结、周复盘、跨项目工作报告、个人投入分析。

### 3.6 UI 只凭形容词直接实现

**导致的问题**

- “纸张质感”容易被实现成规则噪点、星空颗粒或塑料感背景。
- 大字体、大卡片和过度装饰会偏离内容型个人博客。

**最终正确做法**

1. 把参考图拆成颜色、材质、密度、对比度、光泽和负面约束。
2. 先确认视觉方向，再进入代码。
3. 实现后用截图与参考目标对比，不只检查代码是否运行。
4. 把确认后的视觉规则写进项目设计文档。

**适用场景**

- 首页、Hero、背景材质、字体尺度、卡片布局、动效和主题模式。

### 3.7 网络问题先改无关配置

**导致的问题**

- “重新连接 1/5”曾被混同为终端启动脚本问题。
- 实际诊断表明主要问题是 WebSocket 长连接和代理链路。

**最终正确做法**

1. 先检查官方状态和客户端诊断。
2. 分别验证 WebSocket、HTTPS、代理、DNS 和登录状态。
3. 修改配置前先备份，修改后用严格配置检查验证。
4. 不把同时出现的终端报错自动当成网络根因。

**适用场景**

- Codex 重连、代理/VPN、登录异常、网络超时和客户端连接问题。

## 4. 个人风格档案

### 4.1 UI 设计偏好

**稳定偏好**

- 整体克制、内容优先，不像后台系统，也不做模板化营销页。
- 字体偏小、层级清楚，避免大字晃眼和视觉压迫。
- 喜欢 Swiss Modern、Apple / Linear 式秩序感，可结合轻量 Bento，但不接受卡片堆叠。
- 深色不是纯黑：偏哑光深炭黑、石墨灰、再生纸质感。
- 浅色不是纯白：偏暖灰白、象牙白、自然纸张质感。
- 纹理要低对比、不规则、若隐若现；不要规则噪点、星空、明显光泽或塑料感。
- 动效用于表达主题和品牌记忆，不为展示技术而存在。

**工作方式**

- 没有明确视觉方向时，先通过问题、参考图或效果图收敛。
- 用户明确说“执行”后再写代码。
- 视觉实现后要截图对比并记录确认版本。

### 4.2 产品理念

- 个人品牌优先，技术内容作为能力和可信度证明。
- V1 先做可信、可使用、可展示的最小产品，不急着扩展完整平台。
- 前台阅读和个人表达优先于后台、CMS、数据库和复杂运营能力。
- 每一阶段都要有明确目标、非目标、验收条件和结束信号。
- 新问题记录为遗留项，不应随意插入当前任务扩大范围。

### 4.3 交互原则

- 交互状态要明确，用户能知道当前选中、未选、完成、暂停或失败。
- 能自然撤销的选择应支持取消，例如单选项再次点击后取消。
- 页面减少干扰，导航服务于主要入口，详情阅读时可以弱化菜单。
- 复杂业务交互优先遵循现有页面和业务规则，不自行创造新模式。
- 表面限制与提交兜底要同时存在，不能只做视觉禁用。

### 4.4 内容系统偏好

核心闭环：

```text
00-Inbox 捕获
-> 项目推进或问题处理
-> 每周复盘
-> 最多提炼 1-3 个长期节点
-> 放入 02-Knowledge
-> 完成后进入 99-Archive
```

稳定原则：

- 不把聊天流水直接入库。
- 节点要能回答具体问题。
- 项目材料、长期知识、时间线记录和归档各自分工。
- 线程清洗和交接材料跟着项目走，不污染技术沉淀目录。
- 所有有效结论应能反查来源。
- 每条任务、想法和记录都要有下一步或结束状态。

### 4.5 工作流偏好

- 结论先，日常回答简短。
- 需求不清楚时先讨论和提问，不修改文件。
- 明确说“执行、修改、实现”后，才进入操作阶段。
- 默认最小范围修改，不顺手重构或扩展。
- 参考文档、指定页面和用户最新修正优先。
- AGENTS 只放入口和高频约束，流程细则按需读取。
- Dbu Notes 保存自建 Skill 源文件和版本，全局目录负责安装使用。
- 长任务需要阶段检查点，避免持续执行后才发现方向偏差。

## 5. 后续 Codex 可直接遵循的规则

1. 先判断当前请求是讨论、计划、记录、修改、迁移还是归档。
2. 用户没有明确要求执行时，不修改文件。
3. 存在关键歧义时，只问影响执行结果的问题。
4. 没有来源的字段、接口、权限和业务逻辑不要补全。
5. 跨项目迁移必须同时对比来源历史和目标项目现状。
6. 默认最小范围修改，不顺手重构、抽公共组件或扩大任务。
7. 批量改名、迁移和属性补全前，先给预览与影响范围。
8. AI 的完成声明必须绑定测试、差异、运行结果或用户验收。
9. UI 先确认方向和负面约束，再实现并截图对比。
10. 跨线程总结必须检索相关线程和项目证据，不能只看当前对话。
11. 压缩或换线程前生成项目内交接材料，区分确认、待验证和丢弃内容。
12. 知识库只沉淀可复用结论，不保存一次性日志和聊天流水。
13. 用户后续修正优先于早期说法，用户确认优先于 AI 总结。
14. 每项工作结束时记录结果、可复用经验和下一步。

## 6. AGENTS.md 建议

### 用户级 AGENTS.md

**建议加入或保留**

- 结论先、日常回答简短。
- 不捏造；缺少证据时标记待确认。
- 用户后续修正优先。
- 全局记忆只保存跨项目稳定偏好。

**不建议加入**

- 具体项目路径、字段、接口、页面、UI 细节和一次性故障。

### Dbu Notes 项目级 AGENTS.md

现有规则已经覆盖大部分需求。建议只补一个按需入口：

> 复盘 Codex 会话、维护个人风格档案时，读取本文件，并使用会话索引、相关片段和项目证据；更新时去重，不复现原始日志。

不建议把本档案全文复制进 AGENTS，避免入口文件膨胀。

### 业务项目 AGENTS.md

建议加入：

- 没明确要求执行时，先做规划和边界确认。
- 迁移任务必须对比来源历史、目标实现和差异表。
- 未经验证不能声称完成。
- UI 项目引用对应设计规则；业务项目不继承博客视觉偏好。

## 7. 维护方式

- 每周更新一次“执行经验”和“偏好变化”。
- 相同规则合并，不按周重复追加。
- 新结论至少满足一项：重复出现、用户明确确认、解决了重复问题、可跨任务复用。
- 一次性项目细节留在项目复盘，不进入本档案。
- 偏好发生冲突时，以日期更近的用户确认内容为准。
