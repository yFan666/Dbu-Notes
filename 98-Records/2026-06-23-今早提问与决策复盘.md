# 2026-06-23 今早提问与决策复盘

## 事实

今早主要围绕 `Dbu Notes` 的执行规则、目录编号、Obsidian 笔记属性和按需读取规则做整理。

实际发生的事情：

- 复盘了“周内总结不应归档”的错误，并把规则写入最小闭环规则。
- 在 `AGENTS.md` 增加“需求不清晰前先讨论，不修改文件”的规则。
- 将 `02-Knowledge` 下的目录和文件按编号排序：目录从 `00` 开始，文件从 `01` 开始。
- 设计并批量写入 Obsidian 笔记属性，最终核心字段扩展为 `type / status / area / review / created / updated`。
- 将 Git 提交规则和 Obsidian 属性规则拆成按需读取入口，避免每次任务都读取流程细则。

## 提问不足

### 1. 对“本周日常工作总结”的动作边界确认不足

问题：

- 没有先确认“总结”是否等于“整理归档”。
- 在今天是周二的情况下，误把当前周文件迁移到 `98-Records`。

应该先问：

- “这次只是给当前文件补总结，还是要调整它的位置？”
- “周内是否允许归档，还是只在周日处理？”

改进规则：

- 周内处理本周日常工作，只总结当前文件。
- 周日才做周总结、归档判断和下周文件初始化。

### 2. 对“02、03、04 排序”的现状确认不足

问题：

- 当前仓库已经没有顶层 `03`、`04` 目录，但没有第一时间把这个变化作为前置结论讲清楚。
- 用户补充“文件夹从 00 开始，文件从 01 开始”后，才修正排序规则。

应该先问：

- “当前只有 `02-Knowledge`，是否只处理它？”
- “编号规则是目录从 `00` 开始、文件从 `01` 开始吗？”
- “是否需要同步修复 Obsidian 链接和显式路径？”

改进规则：

- 批量重命名前，必须先给出预览表和编号规则，确认后再执行。

### 3. 对 Obsidian 属性字段确认不足

问题：

- 最初只提出 4 个核心字段，没有把 `created / updated` 是否纳入核心字段提前确认清楚。
- 后续用户补充后，才扩展为 6 个字段。

应该先问：

- “核心字段是否只包含 `type / status / area / review`，还是也包含 `created / updated`？”
- “`created` 用文件创建时间、文件修改时间，还是统一处理日期？”
- “已有 frontmatter 是补缺失字段，还是重排字段顺序？”

改进规则：

- 属性批量写入前，必须确认字段集合、默认值、日期来源、已有字段处理方式。

### 4. 对“按需读取规则”的入口设计没有提前抽象

问题：

- Git 规则、Obsidian 属性规则一开始都混在 `AGENTS.md` 或知识节点里。
- 没有第一时间抽象出“流程规则只在对应流程触发时读取”这一层。

应该先问：

- “这条规则是全局每次都读，还是某个流程触发时才读？”
- “是否需要在 `AGENTS.md` 只保留入口，把细则放到独立文件？”

改进规则：

- 只针对某一流程的规则，应在 `AGENTS.md` 建入口，触发时按需读取。

## 决策总结

### 决策 1：AGENTS 只放入口和高频约束

判断：

`AGENTS.md` 应该负责“什么时候读什么”，不应该承载所有流程细节。

结果：

- Git 规则拆到 `02-Knowledge/07-Resources/03-Git-提交规则.md`。
- Obsidian 属性规则保留在 `02-Knowledge/Obsidian-笔记属性.md`。
- `AGENTS.md` 只保留按需读取入口。

### 决策 2：流程规则按需读取

判断：

每次任务都读取所有细则会增加噪音，也容易让任务范围变大。

结果：

- 提交、暂存、推送前才读 Git 规则。
- 新增、编辑、批量补全 Obsidian 属性前才读属性规则。
- 日志、周总结、Inbox、归档、目录迁移前才读最小闭环规则。

### 决策 3：需求不清晰前不改文件

判断：

这类知识库整理任务容易因为一句话产生多种解释，直接修改会放大误差。

结果：

- 在 `AGENTS.md` 中加入：需求没有明确执行路径和处理方式前，只做技术讨论、方案讨论和边界确认。
- 每个问题执行前，需要先向用户确认关键边界、遗漏风险和预期结果。

### 决策 4：笔记属性先做最小可用系统

判断：

属性字段过多会增加维护成本，先保证所有笔记可筛选、可复盘。

结果：

核心字段确定为：

```yaml
type:
status:
area:
review:
created:
updated:
```

已有属性不覆盖，只补缺失字段。

### 决策 5：编号排序用于降低查找成本，不改变内容归属

判断：

编号是导航优化，不是目录迁移。

结果：

- `02-Knowledge` 下目录从 `00` 开始。
- 每个目录里的文件从 `01` 开始。
- `00-Inbox` 不参与本次 `02-Knowledge` 编号规则。
