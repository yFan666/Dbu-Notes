# 最近 10 天高频规则与提问沉淀

日期：2026-06-17
来源：Codex 最近 7-10 天会话记录、全局记忆边界、项目协作提问
用途：作为后续封装项目专门 skill 的原始材料

## 结论

最近高频需求不是“让 AI 多做”，而是让 AI 更像一个稳定的项目协作执行器：少猜、少扩展、按证据闭环、能跨线程接续。

最值得封装成 skill 的不是某个具体项目知识，而是以下 4 类工作流：

1. 需求实现前的范围锁定
2. 参考页面 / 文档驱动实现
3. 防捏造与证据校验
4. 跨线程记录、继续与每日复盘

## 高频规则

### 1. 最小范围修改

反复出现的表达：

- 最小范围更改
- 不要修改其他代码
- 不要做多余检查
- 不要封装公共组件
- 只实现当前需求
- 小型调整不启动完整流程

可沉淀规则：

> 默认只改和当前需求直接相关的内容。除非用户明确要求，不做顺手重构、不抽象公共组件、不扩大影响范围。

适合封装到项目 skill：是。

### 2. 不确定先问，不要擅自决定

反复出现的表达：

- 如果有不确定的问题，向我提问之后再处理
- 不可擅自决定
- 如果我给出的问题不对，你要告诉我
- 不要捏造逻辑
- 参考页面没有的，不要自己编

可沉淀规则：

> 证据不足时停止猜测。必须说明缺哪类证据，并给出最小补充信息；不能用“看起来合理”替代项目事实。

适合封装到项目 skill：强烈适合。

### 3. 参考页面 / 文档优先

反复出现的表达：

- 仿照某个页面
- 参考这个页面逻辑
- 根据需求文档实现
- 和之前某个逻辑一致
- 根据入参和出参做对比

可沉淀规则：

> 实现前先找参考源，明确“照搬什么、差异是什么、哪些不能照搬”。参考源优先级：需求文档 > 指定页面 > 同模块相似页面 > 项目通用写法。

适合封装到项目 skill：是。

### 4. 验证闭环

反复出现的表达：

- 检查是否逻辑闭环
- 检查增删改查
- 样式是否布局合理
- 有没有对不上的情况
- 有没有捏造的情况
- 能跑通需求即可

可沉淀规则：

> 完成后必须回到需求原文逐项对照：功能是否闭环、字段是否一致、权限/打印/导出是否有来源、样式是否破坏布局。

适合封装到项目 skill：是。

### 5. 记忆边界：全局只放协作偏好，项目内容进项目

反复出现的表达：

- 全局规则、全局记忆在哪里
- 只保留回复纪律
- 不需要项目级记忆
- 项目级内容看着乱
- memory_summary 和 MEMORY 有没有效果

可沉淀规则：

> 全局长期记忆只保存跨项目稳定有效的回复纪律、协作偏好和工作方式。项目页面、组件、bug、接口、样式细节只进入项目上下文或 Second Brain 项目记录。

适合封装到项目 skill：不适合放项目 skill；适合保留为全局协作规则。

### 6. 跨线程接续与阶段记录

反复出现的表达：

- 记录目前进程，明天继续
- 继续昨天的任务
- 记住现在修改的明天再继续
- 每日事项整理
- 今天做了什么
- 不要只看当前线程

可沉淀规则：

> 跨线程总结必须读取当前线程、相关项目会话、当天文件变更、项目 git 记录和 Second Brain 记录；不能只凭当前对话判断。

适合封装到项目 skill：适合做“每日复盘 / 任务接续 skill”。

### 7. UI 工作流：先定方向，再实现

反复出现的表达：

- 我不满意，但没有确切想法，你对我提问
- 先出图 / 先分析，不改代码
- 记录这个 UI 作为认可版本
- 只做 UI 规划，不改代码
- Hero、Three.js、动效、视觉隐喻

可沉淀规则：

> UI 不明确时，先用问题收敛风格，再输出可实现描述或设计图；用户确认后再改代码。视觉方向确认前，不进入实现。

适合封装到项目 skill：适合做 blog/UI 子 skill。

## 高频提问类型

### A. 业务功能实现类

典型形式：

```text
修改目标：某个文件/页面
需求：实现某个字段、接口、按钮、弹窗、跳转、校验
约束：参考某页面，最小范围，不要多余改动
验证：检查是否闭环
```

建议模板：

```text
处理这个需求：

目标文件：
参考页面/文档：
需求点：
限制：最小范围修改，不要擅自扩展；不确定先问。
完成后检查：字段、接口、权限、打印/导出、布局、是否有捏造逻辑。
```

### B. 需求整理 / 丢给研发类

典型形式：

```text
根据这个文档整理需求，然后丢给研发
帮我完善需求文档
作为 AI PM 帮我细化需求
```

建议模板：

```text
帮我把这个需求整理成研发可执行文档：

原始材料：
目标页面：
业务目标：
字段/接口/交互：
不确定项：请单独列出，不要替我脑补。
输出：研发任务清单 + 验收标准。
```

### C. 问题排查类

典型形式：

```text
排查为什么还有必填提示
线上偶发问题，无法本地调试
按可能阻碍优先级回复
```

建议模板：

```text
帮我排查这个问题：

现象：
出现条件：
已知限制：是否能本地复现 / 是否只能线上出现
相关文件：
我希望输出：按可能性排序，先给最可能原因和最小验证办法。
```

### D. 规则 / 记忆 / Skill 类

典型形式：

```text
这个规则能否跨线程生效
全局记忆在哪里
哪些内容应该进全局，哪些进项目
安装/研究某个 skill
```

建议模板：

```text
帮我处理一条协作规则：

规则内容：
生效范围：全局 / 当前项目 / 当前线程
是否写入 Second Brain：是 / 否
是否后续封装成 skill：是 / 否
```

### E. UI / 动效类

典型形式：

```text
我没有确切想法，你先问我
先分析图片，不改代码
记录这个 Hero 想法
用 Three.js 实现，但保留 2D 感
```

建议模板：

```text
帮我处理 UI/动效方向：

目标感受：
参考图/页面：
必须保留：
不要出现：
当前阶段：只分析 / 出方案 / 出图 / 改代码
确认后再实现。
```

## 后续可封装的项目 Skill 草案

### 1. `project-requirement-executor`

用途：处理业务页面需求。

核心流程：

1. 读取目标文件、需求文档、参考页面
2. 列出照搬点和差异点
3. 最小范围实现
4. 对照需求逐项验证
5. 输出改动、验证、遗留问题

### 2. `anti-hallucination-implementation`

用途：防止捏造权限、接口、字段、业务逻辑。

核心规则：

1. 没有来源的逻辑不写
2. 参考页面没有就标记为待确认
3. 字段和接口必须从文档、代码或用户输入中来
4. 完成后列出“证据来源”

### 3. `daily-work-review`

用途：每日事项整理和跨线程接续。

核心流程：

1. 查当前线程
2. 查最近会话
3. 查项目 git 和文件变更
4. 按项目归类
5. 输出 Markdown 总结
6. 用户确认后再写入 Second Brain

### 4. `ui-direction-to-implementation`

用途：把模糊 UI 感受转成可实现方案。

核心流程：

1. 先问风格问题
2. 提炼视觉关键词
3. 输出可编程描述
4. 必要时出图
5. 用户确认后再改代码

## 建议优先级

1. 先封装 `anti-hallucination-implementation`
2. 再封装 `project-requirement-executor`
3. 然后封装 `daily-work-review`
4. 最后封装 `ui-direction-to-implementation`

原因：最近最影响效率的不是能力不足，而是“证据边界”和“执行范围”需要更稳定。

## 可直接进入项目 AGENTS 的短规则

```markdown
## 执行边界

- 默认最小范围修改，只处理当前需求直接相关内容。
- 参考页面、需求文档、用户给出的字段和接口优先；没有来源的逻辑不要捏造。
- 不确定时先问，不能擅自决定。
- 完成后按需求逐项检查：字段、接口、权限、打印/导出、布局、验证结果。
- 项目细节写入项目上下文或 Second Brain，不进入全局长期记忆。
```
