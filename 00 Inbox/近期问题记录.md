问题：

```text

1. 如何管理 AGENTS ，什么样的规范作为初版内容
2. 一个完整的项目中，和 ai 如何进行协作
3. 任务的切分，ai 给出的任务清单太过详细，不知道该如果缩减。
4. 线程开的太多，有新的提问就开一个新的线程。
5. 写过的任务或者回答，没法形成复用的

```

## 1. 结论

目前真正的问题不是 Obsidian 怎么存，而是：

> 什么信息值得沉淀，什么信息只是临时记录；什么事情已经结束，什么事情还在占注意力。

先不要搭复杂知识图谱。当前只跑一个最小流程：

```text
日常问题 -> 00 Inbox 临时记录 -> 每周复盘 -> 最多提炼 3 个长期节点 -> 链接到项目/方法/提示词/复盘
```

核心规则：

- 先记录问题，不急着建节点。
- 节点必须能回答一个具体问题。
- 每周最多沉淀 3 个长期节点。
- 每个任务、想法、问题都要有结束信号。

## 2. 当前问题

近期反复出现的问题可以归成 5 类：

| 问题           | 表现             | 本质              |
| ------------ | -------------- | --------------- |
| AGENTS 怎么管理  | 不知道初版规则写到什么程度  | 缺少最小协作规范        |
| 项目中怎么和 AI 协作 | 需求、开发、验收容易混在一起 | 缺少项目协作流程        |
| AI 任务拆分太细    | 清单很长，执行时反而卡住   | 缺少任务粒度标准        |
| 线程太多         | 有新问题就开新线程      | 缺少线程归档和汇总机制     |
| 回答无法复用       | 写过的任务和回答沉不下来   | 缺少从回答到知识节点的转换流程 |

## 3. 可执行方案

### 3.1 每天：只做收集

当天有新问题时，先写进 `00 Inbox`，不要马上拆目录。

记录格式：

```md
## 问题标题

日期：
场景：
问题：
当前答案：
下一步：
状态：进行中 / 已完成 / 暂停 / 放弃 / 待复盘
```

执行标准：

- 标题必须是一个问题，例如“AI 任务拆分太细怎么办”。
- 如果只是聊天灵感，先写“待确认”，不要写成结论。
- 每条记录必须有“下一步”或“状态”，避免悬空。

### 3.2 每个任务结束：补一个结束信号

任务结束时只补 3 行：

```md
结果：
可复用经验：
后续动作：
```

结束信号可以是：

- 已完成
- 暂停
- 归档
- 下一步已记录
- 明确决定不再继续

### 3.3 每周：只沉淀 3 个节点

每周复盘时，从 `00 Inbox` 里挑最多 3 个值得长期保存的内容。

筛选标准：

```text
1. 以后还会不会用到？
2. 能不能解决重复问题？
3. 能不能变成方法、模板或提示词？
4. 是否和当前项目或长期能力有关？
```

满足任意一条，就可以沉淀；一条都不满足，就继续留在 Inbox 或直接归档。

## 4. 放到哪里

按当前仓库结构，不新建复杂分类。

| 内容类型              | 放置位置                         |
| ----------------- | ---------------------------- |
| 临时问题、未整理想法        | `00 Inbox`                   |
| 具体项目、任务、方案、决策     | `01 Projects`                |
| 可复用方法、提示词、AI 协作规则、个人思考、外部资料 | `02 Knowledge` |
| 仍会影响下一步行动的时间线记录、日志、周总结、优先级 | `98 Records` |
| 完成、暂停、过时、冻结内容 | `99 Archive` |

判断不清楚时，先留在 `00 Inbox`。
