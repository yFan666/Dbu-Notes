## 一、项目目标

实时聚合多平台热点数据，支持：

- 按平台分类（微博、知乎、掘金等）
- 按热度排序
- 关键词搜索
- 热点趋势可视化

## 二、核心问题拆解

热点监控工具需要解决的三个问题：

1. **数据从哪来** —— 多平台热点数据源
2. **怎么抓** —— 爬虫 / RSS / 第三方 API
3. **怎么用** —— 聚合、分类、搜索、可视化

## 三、方案设计

### 技术选型

- **MCP（Model Context Protocol）**：让 AI 调用外部工具和数据源，避免每接一个平台就写一遍胶水代码
- **AgentSkill**：把热点聚合流程封装成可复用能力

### 数据获取策略

优先级：官方 API > RSS > 第三方聚合接口 > 爬虫

- 官方 API 最稳但常有限制
- RSS 简单稳定，适合内容型平台
- 第三方聚合接口省事但依赖外部
- 爬虫最灵活但维护成本高、易被封

### 架构思路

```
数据源层（各平台）
   ↓ 适配器（统一输出格式）
处理层（去重、排序、分类）
   ↓
存储层（缓存 + 持久化）
   ↓
展示层（列表 / 搜索 / 可视化）
   ↓
AI 能力层（MCP 调用、Skill 封装）
```

## 四、AI 能力集成

### MCP 使用

通过 MCP 协议接入数据源，AI 可动态调用工具抓取指定平台热点。好处：

- 新增平台只需加一个 MCP tool，不动主流程
- AI 负责编排，人定义边界

### AgentSkill 封装

把"抓取 → 清洗 → 排序 → 返回 Top N"封装成一个 Skill，可被不同 Agent 复用。

> 配置示例（Key 已脱敏）：
> ```text
> OPENROUTER_API_KEY=<已剔除-请从环境变量读取>
> ```
> **正确做法**：API Key 放入 `.env`（加入 `.gitignore`），代码用 `process.env.OPENROUTER_API_KEY` 读取，绝不硬编码。

## 五、经验沉淀

1. **明文密钥是事故源头**：任何 Key 进入 git 仓库前必须剔除，改用环境变量。
2. **MCP 适合"数据源多、接口各异"的场景**：用协议层屏蔽差异。
3. **Skill 封装复用**：重复的抓取-清洗-排序流程应抽象成标准能力。
4. **数据源优先级要明确**：先稳定的，再灵活的，不要一开始就爬虫。
