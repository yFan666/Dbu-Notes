# AI Native 知识库方法论

> 来源：Second Brain（00 Inbox/AI 知识库与工作流记录）
> 整合日期：2026-06-17

---

## 核心目标

建立一个 AI Native 的知识库体系，不记录低价值流水账，而是沉淀：

- 可复用经验
- 决策依据
- 架构思维
- 排障模式
- 通用 Pattern
- AI 可理解的 Context

## 核心原则

### 1. 记录"为什么"，不是"做了什么"

低价值记录（Git 或任务系统已能记录）：
```
今天改了按钮
今天改了接口
今天修了一个 bug
```

高价值记录（决策依据、架构意图、模式、设计哲学）：
```
为什么不用 watchEffect
为什么采用 BFF
为什么动态列采用 Map
为什么不用深拷贝
为什么使用 shallowRef
```

### 2. 知识库是决策系统，不是工作流水账

真正有价值的知识库不是"我做了什么"，而是"**我为什么这样做**"。未来 AI 更需要：决策依据、思维模式、架构原则、通用 Pattern。

### 3. 新应用或新规则不要立刻打断原有工作流

了解到新应用或规则时，先使用原有工作流，不要急着实践。只有在当前事情处理好之后，再利用空闲时间了解和应用。

原因：新工具不一定对当前事情有实质效果；优先保证原有工作的稳定和交付。

## AI 协作与流程规则

工作流要点：
- 只记录每次里程碑的重大决策
- 记录成功或失败，并写明理由
- 每个节点都需要留痕，并标明状态
- 先规划执行，再提供反馈
- 管理任务状态
- 将任务拆成小模块，再分开处理

## 推荐知识模板

### 设计/架构类
```markdown
# 标题
## 背景
## 问题
## 错误方案
## 优化方案
## 为什么这样设计
## 优势
## 缺点
## 适用场景
## 不适用场景
```

### Pattern 模式类
```markdown
# Pattern 名称
## 场景
## 核心思想
## 结构
## 实现方式
## 注意事项
## 示例
```

### 思维模型类
```markdown
# 标题
## 以前的理解
## 新的理解
## 为什么发生改变
## 核心原则
## 后续影响
```

### 排障案例类
```markdown
# 问题标题
## 现象
## 环境
## 问题表现
## 根本原因
## 错误方向
## 解决方案
## 为什么能解决
## 如何避免
## 关联知识
```

## 每周蒸馏方案

平时只记录 Quick Notes：灵感、坑、问题、关键点。

每周统一：分类、提炼、去重、升级为 Pattern。

每周自查问题：
1. 本周重复出现最多的问题是什么？
2. 本周最大的性能问题是什么？
3. 本周最大的认知升级是什么？
4. 哪些代码开始重复？
5. 哪些错误未来还会再犯？

## 知识三层架构

| 层级 | 用途 | 特点 |
|---|---|---|
| **Layer 1：临时层** | 临时记录、草稿、灵感、未整理内容 | 生命周期短、不追求结构、随时可整理或删除 |
| **Layer 2：模式层** | 沉淀可复用经验、通用方案和技术模式 | 高复用、AI 最需要、长期存在 |
| **Layer 3：原则层** | 沉淀思维模型、架构哲学和决策原则 | 生命周期最长、复用率最高、最接近经验 |
