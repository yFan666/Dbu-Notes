# 证据边界项目工作流

## 来源

- 原始沉淀：[[06 2026-06-17-最近10天高频规则与提问沉淀]]

## 适用场景

当任务涉及实现、排查、整理需求、参考页面、跨线程复盘时，使用这套规则。

典型触发词：

- 最小范围
- 参考页面
- 根据文档
- 不要捏造
- 不确定先问
- 检查闭环
- 丢给研发
- 每日复盘
- 跨线程接续

## 核心规则

### 1. 先锁范围，再行动

默认只处理当前需求直接相关内容。

不要顺手重构、抽公共组件、改兄弟页面、做无关格式化，除非用户明确要求或证据显示它属于同一个需求。

### 2. 先找证据，再判断

字段、接口、权限、打印、导出、校验、业务状态和 UI 行为都必须有来源。

证据优先级：

1. 用户明确说明
2. 需求文档
3. 指定参考页面或文件
4. 同模块相似实现
5. 项目已有通用写法
6. 推断，但必须标明是推断

### 3. 不确定先问，不要脑补

证据不足时，不用“看起来合理”代替事实。

如果继续执行会影响结果，就先问用户；如果只是整理草稿，则放到 `待确认`。

### 4. 参考源要拆成照搬点和差异点

参考页面或文档不是整包复制。

每次参考前先拆：

```text
参考源：
照搬点：
差异点：
待确认：
不应照搬：
```

### 5. 结束前必须回到原始需求验证

完成后检查：

- 范围是否保持最小
- 是否存在无来源逻辑
- 字段和接口是否对得上
- 参考行为是否只复制了该复制的部分
- 布局或用户可见行为是否被破坏
- 不确定项是否列为 `待确认`
- 下一步或完结标识是否明确
