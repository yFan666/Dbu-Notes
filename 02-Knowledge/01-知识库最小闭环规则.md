# 知识库最小闭环规则

## 1. 使用手册

当前目标不是搭完整知识图谱，而是先跑通一个稳定闭环：

```text
日常问题
-> 00-Inbox 临时记录
-> 每周复盘
-> 最多沉淀 3 个长期节点
-> 链接到项目 / 方法 / 提示词 / 复盘
-> 标记完结状态
```

判断标准：

- 先让信息能被接住，再考虑分类是否精美。
- 先保证每条记录有下一步，再考虑链接是否丰富。
- 先让每周能沉淀 1-3 个节点，再扩展复杂系统。

## 2. 入库规则

当天新问题先进入 `00-Inbox`，不要马上拆目录。

记录格式：

```markdown
## 问题标题

日期：
场景：
问题：
当前答案：
下一步：
状态：进行中 / 已完成 / 暂停 / 放弃 / 待复盘
```

放置规则：

| 内容类型 | 放置位置 |
| --- | --- |
| 临时问题、未整理想法 | `00-Inbox` |
| 具体项目、任务、方案、决策 | `01-Projects` |
| 可复用方法、提示词、AI 协作规则、个人思考、外部资料 | `02-Knowledge` |
| 仍会影响下一步行动的时间线记录、日志、周总结、优先级 | `98-Records` |
| 完成、暂停、过时、冻结内容 | `99-Archive` |

判断不清楚时，先留在 `00-Inbox`。

线程清洗 / 新线程交接材料按项目放置：

- 某个项目自己的线程清洗总结、交接文档，放在 `01-Projects/项目名/00-线程接力/EPxx-主题/`。
- EP 目录至少保留 `线程清洗总结.md` 或 `新线程交接文档.md`；需要二次加工时再增加 `二次加工记录.md`。
- 具体清洗产物不放项目根目录、不放 `02-Knowledge`、不放 `98-Records`；可复用流程、模板、方法论再提炼到 `02-Knowledge`。

## 3. 节点命名与链接规则

节点必须能回答一个具体问题。

优先使用问题式标题：

- `AI 任务拆分太细怎么办`
- `线程太多如何管理`
- `AI + Obsidian 信息如何入库`

只保留 4 类链接：

```text
问题 -> 方法
方法 -> 项目
复盘 -> 节点
提示词 -> 使用场景
```

不要为了知识图谱而知识图谱。没有明确用途的链接先不加。

## 4. 复盘与总结流程

每天只做收集。

周内处理“本周日常工作”时，只对当前文件做简短总结，不移动、不归档、不新建正式记录目录。

周日才做周总结、归档判断和下周文件初始化。

每个任务结束时补 3 行：

```markdown
结果：
可复用经验：
后续动作：
```

每周从 `00-Inbox` 最多挑 3 个长期节点。

筛选标准：

```text
1. 以后还会不会用到？
2. 能不能解决重复问题？
3. 能不能变成方法、模板或提示词？
4. 是否和当前项目或长期能力有关？
```

复盘格式：

```markdown
事实：

判断：

下一步：
```

每次复盘都要有下一步。没有下一步，就明确标记为完成、暂停、归档或不再追踪。

## 6. 自建 Skill 管理规则

自建 skill 统一使用 `dbu-` 前缀，和网上拉取、系统内置、第三方 skill 区分。

命名示例：

```text
dbu-evidence-bounded-project-workflow
dbu-daily-work-review
dbu-inbox-distiller
dbu-project-requirement-executor
```

管理原则：

- `C:\Dbu Notes\.agents\skills` 是源文件目录，用来版本管理和后续优化。
- `C:\Users\Administrator\.agents\skills` 是全局安装目录，用来让 Codex 在其他项目中识别和使用。
- 优化 skill 时，先改 Dbu Notes 中的源文件，再同步复制到全局目录。
- 全局目录不作为唯一来源，避免多台电脑或多个项目之间版本不一致。

一句话规则：

> Dbu Notes 管版本，全局目录管使用；自建 skill 用 `dbu-` 前缀。
