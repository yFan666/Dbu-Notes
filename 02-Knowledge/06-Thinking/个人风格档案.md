# 个人风格档案

来源：[[98-Records/2026.06.18-2026.06.24-Codex-会话复盘与个人风格档案#4. 个人风格档案|近期 Codex 会话复盘]]

## UI 设计偏好

### 稳定偏好

- 整体克制、内容优先，不像后台系统，也不做模板化营销页。
- 字体偏小、层级清楚，避免大字晃眼和视觉压迫。
- 喜欢 Swiss Modern、Apple / Linear 式秩序感，可结合轻量 Bento，但不接受卡片堆叠。
- 深色不是纯黑：偏哑光深炭黑、石墨灰、再生纸质感。
- 浅色不是纯白：偏暖灰白、象牙白、自然纸张质感。
- 纹理要低对比、不规则、若隐若现；不要规则噪点、星空、明显光泽或塑料感。
- 动效用于表达主题和品牌记忆，不为展示技术而存在。

### 工作方式

- 没有明确视觉方向时，先通过问题、参考图或效果图收敛。
- 用户明确说“执行”后再写代码。
- 视觉实现后要截图对比并记录确认版本。

## 产品理念

- 个人品牌优先，技术内容作为能力和可信度证明。
- V1 先做可信、可使用、可展示的最小产品，不急着扩展完整平台。
- 前台阅读和个人表达优先于后台、CMS、数据库和复杂运营能力。
- 每一阶段都要有明确目标、非目标、验收条件和结束信号。
- 新问题记录为遗留项，不应随意插入当前任务扩大范围。

## 交互原则

- 交互状态要明确，用户能知道当前选中、未选、完成、暂停或失败。
- 能自然撤销的选择应支持取消，例如单选项再次点击后取消。
- 页面减少干扰，导航服务于主要入口，详情阅读时可以弱化菜单。
- 复杂业务交互优先遵循现有页面和业务规则，不自行创造新模式。
- 表面限制与提交兜底要同时存在，不能只做视觉禁用。

## 内容系统偏好

核心闭环：

```text
00-Inbox 捕获
-> 项目推进或问题处理
-> 每周复盘
-> 最多提炼 1-3 个长期节点
-> 放入 02-Knowledge
-> 完成后进入 99-Archive
```

稳定原则：

- 不把聊天流水直接入库。
- 节点要能回答具体问题。
- 项目材料、长期知识、时间线记录和归档各自分工。
- 线程清洗和交接材料跟着项目走，不污染技术沉淀目录。
- 所有有效结论应能反查来源。
- 每条任务、想法和记录都要有下一步或结束状态。

## 工作流偏好

- 结论先，日常回答简短。
- 需求不清楚时先讨论和提问，不修改文件。
- 明确说“执行、修改、实现”后，才进入操作阶段。
- 默认最小范围修改，不顺手重构或扩展。
- 参考文档、指定页面和用户最新修正优先。
- AGENTS 只放入口和高频约束，流程细则按需读取。
- Dbu Notes 保存自建 Skill 源文件和版本，全局目录负责安装使用。
- 长任务需要阶段检查点，避免持续执行后才发现方向偏差。
