# Smarthos 微前端基座面试追问清单

> 来源：Second Brain（面试准备/微前端基座）
> 整合日期：2026-06-18
> 场景：HIS 平台微前端基座面试准备

---

## 1. 项目亮点表述

### 简历写法

**复杂路由与状态协同设计**

参与 HIS 平台微前端基座建设，将后端动态菜单转化为基座侧路由索引，通过 `microApp.router`、`setData/dispatch` 实现主子应用双向通信、跨应用跳转、标签页缓存清理和权限菜单联动，降低多业务模块独立部署下的导航与状态同步复杂度。

### 口头扩展版

基座不直接承载所有业务页面，而是统一维护登录、菜单、权限、标签页、应用切换和消息分发。实际业务页面由各子应用自己的 router 渲染。基座通过 `micro-app` 加载当前选中的子应用，并通过 `microRouterPush` 统一判断目标页面属于基座、iframe、WinForm 还是某个子应用。

---

## 2. 微前端选型

**可能被问**：为什么用 micro-app 不用 qiankun/iframe/Module Federation？核心机制是什么？为什么开启 iframe 模式？keep-alive、keep-router-state 解决什么问题？

**回答要点**：项目需要多个 HIS 子系统独立开发和部署，同时由基座统一登录、菜单、权限和标签页。micro-app 接入成本低，子应用只需提供生命周期和通信能力。开启 iframe 是为了增强隔离，减少样式、全局变量和第三方库污染。keep-alive 保留子应用实例避免频繁销毁重建；keep-router-state 保留子应用内部路由状态。

**权衡**：iframe 隔离强，但带来通信、弹窗层级、性能和状态共享成本。医疗业务系统第三方控件、历史包袱多，隔离优先级更高。

---

## 3. 子应用接入

子应用列表在 `packages/smarthos-base-admin/src/config/amicroList.ts`，每个配置 `name` 和 `url`。开发环境按端口区分（8901、8902），生产环境用 `window.location.origin + 子应用路径`。

当前加载哪个子应用由 Pinia 的 `seleApp.perms` 决定。子应用需支持 `window.mount`、`window.unmount` 生命周期，`window.__MICRO_APP_ENVIRONMENT__` 存在时由基座挂载。

---

## 4. 路由处理

基座 Vue Router 负责登录页、首页、错误页、外壳和标签页。子应用 router 负责真实业务页面。

后端菜单被基座转成路由索引和菜单数据，业务页面不由基座组件渲染，所以这些业务路由本质上是"索引路由"或"假路由"。

统一跳转入口 `microRouterPush`：
- 基座页面 → 基座 `router.push`
- iframe 页面 → 动态添加 iframe 路由
- WinForm 页面 → websocket 打开客户端窗体
- 当前子应用页面 → `microApp.router.push`
- 其他子应用页面 → 先切换 `seleApp`，缓存目标 path，等新子应用 mounted/aftershow 后再跳

**关键设计点**：跨应用跳转不能直接 push，目标子应用可能还没加载完成。基座把目标路径暂存在 `changeAppPath`，等 `<micro-app>` 的 `mounted` 或 `aftershow` 触发后再跳。

## 5. 主子应用通信

```ts
// 基座 -> 子应用
microApp.setData(appName, { type: "jump", path })

// 子应用监听
window.microApp.addDataListener(data => {
  if (data.type === "jump") router.push(data.path)
})

// 子应用 -> 基座
window.microApp.dispatch({ type: "appChange", id, path })
// 基座通过 <micro-app @datachange="handleDataChange" /> 接收
```

常见消息类型：`jump`（跳转）、`appChange`（切换子应用）、`delTag`（删除标签页）、`keepAlive`（清理子应用缓存）、`code === 702`（登录失效）。

> 工程判断：通信协议最好保持结构化和集中处理，否则子应用随意操作基座状态会导致状态来源不清晰。

---

## 6. 标签页和缓存

基座维护统一标签页 `multiTags`。关闭标签时，如果页面配置了 `keepAlive`，基座按 `appName` 分组，通过 `microApp.setData(appName, { type: "keepAlive", deleteTags })` 通知子应用清理内部缓存。

> micro-app 的 `keep-alive` 保留子应用实例；Vue 的 `KeepAlive` 保留子应用内部页面组件。两者不是一回事，关闭基座标签页时仍需通知子应用清理内部组件缓存。

---

## 7. 权限与菜单

登录后基座获取用户应用列表和菜单树。`appMenu` 表示用户可访问的应用，`wholeMenus` 表示当前展示菜单，`allRouterList` 是扁平化后的路由索引。

基座负责统一菜单展示、标签页和入口权限。按钮权限通过接口获取后存到 session，由权限组件或指令判断。子应用如有更细粒度业务权限，可在内部继续校验，但入口、导航和页签层面权限由基座统一管理。

---

## 8. 隔离、部署和排障

通过 `iframe` 模式增强隔离，降低样式、全局变量和第三方库冲突。加载失败通过 `microApp.start` 的 lifecycle `error` 捕获并提示。

**排查白屏优先检查**：
- 子应用 URL 是否能访问
- 子应用 base/publicPath 是否正确
- `window.mount/window.unmount` 是否暴露
- 子应用是否识别微前端环境
- micro-app lifecycle 是否报错
- iframe 内控制台是否有资源 404 或路由 404

## 9. 高阶追问

**关键点**：回答不要只说 API 名称，要说明问题背景。

- `changeAppPath`：跨应用跳转存在加载时序，新应用未 mounted 时无法可靠执行内部路由跳转，所以先缓存目标路径，等 mounted/aftershow 后再跳。
- `location.reload()`：承认不是最优雅。优点是简单可靠、能重置基座和当前子应用状态；缺点是体验和性能一般。更好方向是用响应式切换 `seleApp` 让 `<micro-app>` 重新加载，同时用状态机保证目标路径跳转时序。

---

## 10. 一分钟回答模板

基座维护菜单、权限、标签页和路由索引，真实业务页面由子应用自己的 router 渲染。子应用地址在 `amicroList` 中按环境配置，当前应用由 `seleApp.perms` 决定，基座通过 `<micro-app>` 加载并开启 `iframe`、`keep-alive`、`keep-router-state`。

路由上，基座把后端菜单转成路由索引用于菜单展示、权限和标签页，实际业务页面交给子应用 router。统一跳转入口 `microRouterPush` 区分基座页面、iframe、WinForm、子应用页面；跨子应用先切换 app 并缓存目标 path，等新子应用 mounted 后再跳。

通信上，基座用 `microApp.setData` 发 `jump`、`keepAlive` 等指令，子应用用 `dispatch/forceDispatch` 回传 `appChange`、`delTag`、登录失效等事件，基座在 `datachange` 中统一处理。

---

## 11. 最需要背熟的三句话

1. 基座维护的是菜单、权限、标签页和路由索引，真实业务页面由子应用自己的 router 渲染。
2. `microRouterPush` 是统一跳转入口，负责区分基座页面、iframe、WinForm、当前子应用和跨子应用跳转。
3. 主子通信通过 `setData/addDataListener` 和 `dispatch/datachange` 完成，跨应用跳转通过缓存目标路径解决加载时序问题。
