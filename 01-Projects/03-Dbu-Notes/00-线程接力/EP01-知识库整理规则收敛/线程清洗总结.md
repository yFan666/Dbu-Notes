# 线程清洗总结：知识库整理规则收敛

## 1. 线程核心目的

本线程的核心目的，是把 Dbu Notes 从“内容很多、规则分散”的状态，收敛成一个可执行的最小闭环系统。

重点不是继续扩展分类，而是明确：什么内容先进入 Inbox、什么内容进入项目、哪些规则需要每次读取、哪些规则只在特定流程触发时读取，并减少 AI 在需求不清时直接动文件带来的误操作。

## 4. 可复用经验 / 方法论【中高可信】

- 【用户确认】规则不应全部塞进 AGENTS；AGENTS 更适合作为入口，具体流程规则按需读取。
- 【AI总结】知识库最小闭环优先级是：先接住信息，再整理分类；先记录下一步，再扩展链接；先形成每周少量沉淀，再搭复杂系统。
- 【AI总结】流程型规则适合拆成独立文档，例如 Git 规则只在提交、暂存、推送时读取，Obsidian 属性规则只在新增、编辑、批量补全属性时读取。
- 【AI总结】工作日的日常工作文件只做当前周记录和简短总结，不做归档；归档判断应集中到周日。
- 【AI总结】批量整理目录和文件编号前，需要先确认范围、排序规则、链接影响和不处理的目录，避免把结构整理变成大规模误迁移。
- 【AI总结】为已有 Markdown 增加 Obsidian 属性时，应优先保留已有 frontmatter，只补缺失字段，避免覆盖有语义的旧属性。
- 【AI总结】线程审计应优先保留用户原话和用户确认结论；没有来源的 AI 扩展只能标记为待验证，不能当作确定规则沉淀。

## 7. 最终压缩版总结

本线程真正沉淀的核心是：Dbu Notes 要从复杂分类收敛到最小闭环，优先保证信息能进入 Inbox、有下一步、有完结标识，再做少量长期沉淀。用户确认了目录减法、文件编号、Obsidian 属性、流程规则按需读取、工作日只总结不归档、周日再处理归档和下周初始化。后续最重要的是把 AGENTS 作为入口，把 Git、属性、线程审计、日常总结等流程规则拆成可按需读取的文档，并避免在需求不清时直接改文件。
