## 二、技术栈

### 前端

- **Next.js 14**：App Router 架构，支持服务端渲染和 API Routes
- **TypeScript 5.4**：强类型，提升代码质量和可维护性
- **React 18**：并发特性和 Suspense 提升体验
- **Zustand**：轻量状态管理，替代 Redux 降低复杂度
- **Ant Design + Radix UI**：组件库组合，平衡美观和可访问性

### 后端

- **Prisma 6.18**：现代化 ORM，类型安全且支持 Migration
- **MySQL**：关系型数据库，满足复杂查询
- **Redis (ioredis)**：高性能缓存，支持多种数据结构
- **Winston**：企业级日志，支持轮转和结构化输出
- **Zod**：运行时类型验证，保障 API 数据安全

### 其他

- **Docker**：使用
- **NestJS API**：复杂接口规范
- **monorepo**：多应用共享包

## 三、首页最近文章（核心闭环）

> 目标一句话：首页 = 可读、可点、可持续更新

### 任务清单

- [ ] `P0` 首页展示 3 块内容
- [ ] `P0` 最近文章列表：展示最近 5~8 篇（标题、日期、摘要、标签、阅读入口）
- [ ] `P1` 空状态与容错：无文章时给引导（"去创建第一篇文章"），摘要缺失时兜底

### 可直接做的一个小闭环

1. **定义首页"最近文章"数据结构**
   字段：`id/slug`、`title`、`summary`、`cover?`、`tags[]`、`publishedAt`
2. **后端加"最近文章"接口**
   `GET /posts/recent?limit=6`，按发布时间倒序，默认过滤草稿
3. **前端首页接接口并渲染列表**
   先做静态卡片 + 跳转，保证可用
4. **补空状态与加载态**
   无数据 / 慢网下也不难看
5. **验收**
   新增文章后首页自动更新，点击可进详情页，移动端不破版

### 完成标准

- 首页能看到最近文章
- 每篇有关键信息
- 空状态可用
- 通过一次完整预览流程

## 五、项目初始化顺序建议

不要一开始就追求"功能很多"，更好的顺序：

```
先定规则 → 再定目录 → 再定内容模型 → 再定首页/文章页 → 最后扩展搜索/分类/标签/归档/评论等能力
```

> AGENTS.md > 架构目录 > 文章模型 > 页面实现 > 功能增强

### 应优先完善的 5 个规则（按重要程度）

| 优先级 | 规则         | 为什么重要       |
| --- | ---------- | ----------- |
| 1   | 目录结构规则     | 防止 AI 乱放文件  |
| 2   | 技术栈规则      | 防止 AI 擅自换方案 |
| 3   | 内容模型规则     | 博客项目的核心     |
| 4   | Codex 工作规则 | 防止它大范围乱改    |
| 5   | UI 视觉规则    | 保证博客有统一审美   |

> 关键认知：专家的关键是 —— 任务必须有目标、范围、约束、验收标准、执行流程

## 六、dbu-blog 复盘 Prompt

> 可复用模板：审查 AI 协作质量与效率，沉淀后续优化建议

### 分析范围

1. 检查所有与 `dbu-blog` 相关的 AI 会话记录、缓存文件、项目产物、提交记录和中间文件
2. 按任务梳理做过的事：需求提出、代码修改、调试、测试、部署、文档整理
3. 分析每个任务执行结果：是否完成、质量、是否返工、是否低效
4. 审查每个任务的用户提示词、目标定义、约束条件、上下文提供、任务拆解
5. 对比不同任务的推进方式，找出常见优点和问题
6. 总结 AI 协作表现：哪里帮得好，哪里走偏，哪里需要更明确的信息
7. 提炼后续更高效的提示词写法和任务拆解方法

### 分析重点

不只总结"做了什么"，要重点评价"质量"和"协作效率"：

1. 目标是否明确
2. 约束是否完整
3. 是否提前说明技术栈、目录结构、边界条件
4. 是否让 AI 先读代码再动手
5. 是否要求 AI 做验证
6. 是否过早进入实现，导致返工
7. 任务范围是否过大、拆解是否不合理
8. 提示词是否表达含糊、跳步、默认前提不清
9. AI 是否理解偏差或执行路径不稳
10. 是否存在可沉淀为固定工作流或项目规则的经验

### 评分维度（1-5 分）

- 目标清晰度
- 上下文完整度
- 任务拆解合理度
- 执行效率
- 最终结果质量
- 返工风险：低 / 中 / 高
