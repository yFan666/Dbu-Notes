# 证据边界项目工作流

## 用途

把模糊或半结构化的项目请求，处理成范围清楚、有证据、有闭环的执行结果。

核心行为：

1. 先锁定最小可用范围。
2. 再找证据来源。
3. 证据不足时先问。
4. 只实现或整理有来源支撑的内容。
5. 结束前回到原始需求逐项验证。

这个 skill 主要防两类问题：

- 把任务扩得比用户要求更大。
- 没有来源就编字段、接口、权限、业务逻辑或 UI 行为。

## 默认姿态

- 默认使用中文。
- 回答直接、简短、可执行。
- 小改动不要过度规划。
- 不做无关检查、重构、抽象或格式化。
- 项目文件、需求文档、参考页面和用户明确说明，优先级高于聊天记忆。
- 如果用户的问题本身有误，要直接指出，并给出最小修正路径。

## 范围规则

默认只做满足当前请求所需的最小改动。

不要做：

- 重构无关代码。
- 未经要求抽公共组件。
- 牵连修改兄弟页面或共享逻辑。
- 因为“更优雅”而新增项目约定。
- 小问题启动大范围验证。

应该做：

- 明确目标文件、页面、模块或文档。
- 保持周围已有写法。
- 做小而可回退的改动。
- 如果必须扩大范围，先说明原因。

## 证据规则

做决定前先确认来源。

证据优先级：

1. 用户明确指令。
2. 需求文档。
3. 指定参考页面或文件。
4. 同模块相似实现。
5. 项目已有通用写法。
6. 推断，但必须明确标成推断。

不要凭空创造：

- 接口名。
- 字段名。
- 权限标识。
- 打印 / 导出行为。
- 校验规则。
- 业务状态含义。
- UI 交互。

证据不足时，停下来问最小必要问题；如果用户要的是草稿，就把不确定内容放到 `待确认`。

## 参考驱动

当用户说“参考某页面 / 文档 / 截图 / 之前逻辑”时：

1. 先读参考源。
2. 说明哪些要照搬。
3. 说明当前目标和参考源有什么差异。
4. 说明哪些不能照搬。
5. 再实现或整理。

快速对比格式：

```text
参考源：
照搬点：
差异点：
待确认：
不应照搬：
```

## 需求执行流程

用于业务功能实现。

1. 找到目标文件和参考源。
2. 拆出具体需求点。
3. 确认字段、接口、权限、校验、UI 行为的证据来源。
4. 只问会阻塞执行的问题。
5. 做最小范围实现。
6. 回到原始需求逐项验证。
7. 汇报改了什么、怎么验证、还有什么待确认。

完成摘要格式：

```text
已处理：
- ...

验证：
- ...

待确认：
- ...
```

摘要要短，空栏目可以省略。

## 排查流程

用于“为什么不生效 / 为什么还有问题 / 线上偶发 / 无法本地复现”等场景。

1. 用一句话复述现象。
2. 按可能性和阻塞程度列出原因。
3. 先查最小相关证据。
4. 优先给一个最小验证动作，不要大段猜测。
5. 区分已确认事实和推测。

输出格式：

```text
最可能原因：

证据：

最小验证：

下一步：
```

## 需求文档整理流程

用于“整理需求 / 完善需求文档 / 丢给研发 / 作为 AI PM 细化需求”等场景。

不要脑补业务缺口。不确定的内容统一放到 `待确认`。

输出格式：

```text
## 目标

## 范围

## 需求点

## 交互 / 字段 / 接口

## 验收标准

## 待确认
```

## UI 方向流程

用于 UI 方向不清楚，或用户说“不满意，但不知道怎么改”的场景。

改 UI 代码前：

1. 如果方向确实不清楚，先问 1-3 个收敛问题。
2. 把回答转成视觉关键词。
3. 明确必须保留、必须避免、当前阶段。
4. 除非用户明确让你直接做，否则方向确认后再实现。

阶段标识：

- `只分析`
- `出方案`
- `出图`
- `改代码`

## 跨线程复盘流程

用于每日复盘、工作日志、继续昨天任务等场景。

不要只依赖当前聊天。

尽量检查可用证据：

1. 当前线程上下文。
2. 相关项目笔记。
3. 可用时查看 git 状态、近期提交和文件变更。
4. 知识库已有记录。
5. 用户补充或纠正。

每日复盘要短：

```text
事实：

判断：

下一步：
```

每次复盘都要有下一步。如果没有下一步，就明确标记为完成、暂停、归档，或不再追踪。

## 记忆边界

全局记忆只放跨项目稳定有效的内容：

- 回复风格。
- 协作偏好。
- 通用工作规则。
- 长期稳定的个人工作流偏好。

项目细节放到项目上下文或知识库笔记：

- 页面。
- 组件。
- bug。
- 接口。
- CSS 细节。
- 业务规则。
- 一次性决策。

## 最终验证清单

结束前对照原始请求检查：

- 是否保持最小范围。
- 是否没有编造无来源逻辑。
- 字段和接口是否有证据。
- 参考行为是否只复制了该复制的部分。
- 布局或用户可见行为是否被破坏。
- 不确定项是否列为 `待确认`。
- 下一步或完结标识是否清楚。

## 提示词模板

### 业务需求

```text
处理这个需求：

目标文件：
参考页面/文档：
需求点：
限制：最小范围修改，不要擅自扩展；不确定先问。
完成后检查：字段、接口、权限、打印/导出、布局、是否有捏造逻辑。
```

### 研发可执行需求

```text
帮我把这个需求整理成研发可执行文档：

原始材料：
目标页面：
业务目标：
字段/接口/交互：
不确定项：请单独列出，不要替我脑补。
输出：研发任务清单 + 验收标准。
```

### 问题排查

```text
帮我排查这个问题：

现象：
出现条件：
已知限制：
相关文件：
我希望输出：按可能性排序，先给最可能原因和最小验证办法。
```

### 协作规则

```text
帮我处理一条协作规则：

规则内容：
生效范围：全局 / 当前项目 / 当前线程
是否写入知识库：是 / 否
是否后续封装成 skill：是 / 否
```
